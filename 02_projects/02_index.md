# 📊 02_projects – Projetos Ativos em Execução

## Bem-vindo ao Espaço de Trabalho de Projetos Vivos

Este diretório é diferente de tudo que você viu até agora. Não é um diretório de **regras** (como `/00_meta/framework_rules/`). Não é um diretório de **templates virgens** (como `/05_templates/Docs_complementares/`). Este é o diretório de **projetos reais em andamento**.

Aqui você encontra 8 arquivos de documentação que foram copiados de templates, preenchidos com dados reais, e estão sendo ativamente usados. Estes documentos não são teóricos. São vivos. São utilizados por equipes reais trabalhando em projetos reais.

A estrutura reflete o workflow real: você começou em descoberta (questionário de marca), evoluiu para planejamento (briefing e documento de planejamento), e está agora em execução (com recursos de planejamento de projeto e mensagens estruturadas).

**Total de Arquivos Ativos:** 8 | **Última Atualização:** 2 de Dezembro de 2025

---

## Entendendo o Ciclo Vida de um Projeto

Para compreender por que estes arquivos existem e em qual ordem estão, pense no ciclo de vida típico de projeto:

1. **Descoberta:** Você faz muitas perguntas. O que o cliente faz? Por quê? Para quem? Quais são os desafios? (Arquivo: Brand Profile Questionnaire)

2. **Compreensão de Contexto:** Você pesquisa o mercado onde o cliente opera. Quem são os concorrentes? Qual é o tamanho do mercado? Quais são as tendências? (Arquivo: pesquisa_de_mercado)

3. **Briefing:** Você sintetiza descoberta e pesquisa em um briefing executivo que alinha stakeholders no "por quê" e "o quê" antes de discutir "como". (Arquivo: project_brief_template)

4. **Planejamento:** Uma vez que briefing está aprovado, você desce em profundidade: qual é a estratégia? Quais são os objetivos SMART? Qual é o cronograma detalhado? Qual é o orçamento? (Arquivo: MODELO_DOCUMENTO_PLANEJAMENTO_PROJETO)

5. **Se for campanha:** Você adiciona camadas de marketing: qual é a narrativa central? Quais são os pilares de mensagem? Quais são as métricas de sucesso? (Arquivos: Documento de Planejamento de Campanha, Framework de Mensagens, KPIs)

6. **Execução:** Agora você está implementando. Você faz o trabalho real enquanto monitora progresso contra as métricas definidas.

## Os Oito Arquivos Ativos Explicados

### **Grupo 1: Descoberta e Compreensão**

Estes dois primeiros arquivos representam fase de descoberta: você compreendendo o cliente e o mercado.

---

#### **1. questionário_marcas/Brand Profile Questionnaire.md – Descoberta Profunda do Cliente**

Este arquivo contém a descoberta inicial com cliente Aura Brews – marca de café sustentável.

O questionário não é superficial. Você vê seções para visão geral da empresa, modelo de negócio, público-alvo segmentado, desafios atuais, aspirações futuras, e oportunidades de crescimento. O cliente (Elena Rodriguez, Fundadora e CEO) respondeu profundamente cada pergunta.

**Por que este arquivo está aqui:**
Quando você começa projeto, primeira coisa que você faz é compreender cliente profundamente. Você não pode estruturar estratégia se não compreende negócio cliente. Este questionário captura esse entendimento inicial.

**Estágio do projeto:** Semana 1 de projeto (pós-assinatura, pré-planejamento)

**Como ele é usado:** 
- Compartilhado em reuniões de kickoff
- Consultado sempre que você precisa se lembrar de algo sobre o cliente
- Base para todas as decisões estratégicas futuras
- Referência quando cliente questiona por que você recomendou algo

**Exemplo real:** Brand é Aura Brews, café de sourcing ético focado em ESG. Público-alvo principal são "connoisseurs conscientes" – pessoas que valorizam qualidade E impacto ambiental.

---

#### **2. pesquisa_de_mercado/pesquisa_de_mercado.md – Análise de Contexto de Mercado**

Este arquivo documenta pesquisa de mercado da Aura Brews usando metodologia de Philip Kotler.

O arquivo estrutura análise sistemática: tamanho de indústria, tendências de mercado, análise competitiva (quem são os concorrentes de café sustentável?), análise de consumidor (por que pessoas compram café? O que motivações?), análise de oportunidades (qual é o espaço não preenchido?).

**Por que este arquivo está aqui:**
Você não pode desenvolver estratégia em vácuo. Você precisa compreender o mercado onde seu cliente opera. Qual é o tamanho da oportunidade? Há crescimento? Há espaço para novo player? Este arquivo captura esse entendimento.

**Estágio do projeto:** Semana 2-3 (paralelo com questionário)

**Como ele é usado:**
- Apresentado em reunião de revisão estratégica com cliente
- Referenciado em documento de planejamento para justificar posicionamento
- Base para recomendações de estratégia
- Validação de que você realmente compreende o mercado

**Exemplo real:** Pesquisa descobre que mercado de café sustentável está crescendo 15% anualmente, há 3 players principais, e há abertura para marca mais focada em storytelling de origem. Isso valida estratégia de foco em ESG.

### **Grupo 2: Estruturação e Planejamento**

Os próximos arquivos representam transição de descoberta para planejamento: você transformando insights em plano.

---

#### **3. Briefing_projeto/project_brief_template.md – O Ponto de Alinhamento**

Este arquivo é o "briefing executivo" que alinha todos no projeto: o que fazemos? Por quê? Para quem? Quando? Com qual orçamento?

Note que é chamado "template" mas é preenchido com dados reais: Projeto Alpha – Implementação de Dashboard de Análise em Tempo Real. Cliente é Acme Corporation. Budget é R$ 250.000. Project Manager é Jane Doe. Timeline é maio a agosto de 2025.

**Por que este arquivo está aqui:**
Briefing é hora que você "congela" entendimento. Todos os stakeholders (cliente, seu gerente, sua equipe) revisa, faz perguntas, e concorda. Uma vez que briefing está aprovado, você tem permissão para ir a próximo passo. Sem briefing aprovado, você ainda está em discovery.

**Estágio do projeto:** Fim da semana 3 / começo da semana 4

**Como ele é usado:**
- Impresso e assinado por stakeholders principais (cliente, PM, gerente)
- Referência quando escopo é questionado ("está no briefing? Então vamos fazer. Não está no briefing? Não vamos fazer.")
- Base para comunicação com equipe
- Referência em contrato/proposta

**Exemplo real:** Aura Brews quer campanha ESG. O briefing diria: "Objetivo é aumentar brand awareness entre consumidores conscientes com foco em ESG. Métrica é 40% aumento em engagement social. Timeline é 6 meses. Budget é X."

### **Grupo 3: Planejamento Detalhado de Projeto**

Os próximos quatros arquivos detalham diferentes aspectos do planejamento de execução.

---

#### **4. Planejamento_projeto/MODELO_DOCUMENTO_PLANEJAMENTO_PROJETO.md – Planejamento Geral**

Este é documento técnico completo de planejamento. Contém estrutura completa: executive summary, background, objetivos, metodologia, fases detalhadas, cronograma, orçamento por fase, responsabilidades por papel, riscos identificados com planos de mitigação, premissas, critérios de sucesso.

**Por que este arquivo está aqui:**
Se briefing é "o que vamos fazer", planejamento detalhado é "como vamos fazer em detalhe". Briefing é 2 páginas. Planejamento é 15+ páginas. Briefing é para stakeholders executivos. Planejamento é para equipe de projeto.

**Estágio do projeto:** Semana 4-6

**Como ele é usado:**
- Base para criar cronograma detalhado em ferramentas de PM (Asana, Jira, etc.)
- Referência para priorização de trabalho
- Recurso para gestão de riscos
- Documento que você revisa mensalmente conforme projeto evolui

---

#### **5. Planejamento_projeto/MODELO_DOCUMENTO_REQUISITOS_PROJETO.md – Especificação de Requisitos**

Este arquivo detalha todos os requisitos técnicos, funcionais e não-funcionais do projeto. Define exatamente o que deve ser entregue e como deve funcionar.

**Por que este arquivo está aqui:**
Requisitos claros são essenciais para evitar ambiguidades. Este documento funciona como "contrato técnico" entre stakeholders sobre o que será entregue.

**Estágio do projeto:** Semana 5-7

**Como ele é usado:**
- Validação com cliente do que será implementado
- Referência para equipe técnica durante desenvolvimento
- Base para testes e garantia de qualidade
- Documento de rastreabilidade de requisitos

---

#### **6. Planejamento_projeto/protocolo_tomada_decisao_projeto.md – Protocolo de Decisões**

Este arquivo estabelece protocolo padronizado para como decisões de projeto serão tomadas, documentadas e comunicadas.

**Por que este arquivo está aqui:**
Decisões mal documentadas causam confusão, retrabalho e conflitos. Este protocolo garante consistência e rastreabilidade.

**Estágio do projeto:** Semana 4

**Como ele é usado:**
- Framework para documentar todas as decisões importantes
- Referência quando alguém questiona uma decisão anterior
- Comunicação com stakeholders sobre processo decisório
- Registro histórico do projeto

---

#### **7. Planejamento_projeto/template_plano_comunicacao_projeto.md – Plano de Comunicação**

Este arquivo detalha como comunicação será gerenciada durante o projeto: quem comunica? O quê? Quando? Para quem? Em qual canal?

**Por que este arquivo está aqui:**
Comunicação deficiente é causa raiz de muitos problemas em projetos. Este plano garante que todos estejam informados de forma consistente.

**Estágio do projeto:** Semana 5

**Como ele é usado:**
- Template para reuniões, atualizações e reportes
- Cronograma de comunicação com cliente
- Identificação de stakeholders e suas necessidades de informação
- Controle de versões e distribuição de documentos

---

### **Grupo 4: Execução e Mensagens**

O último arquivo reflete a camada de execução quando projeto envolve marketing/campanha.

---

#### **8. Planejamento_campanha/Framework de Mensagens da Campanha.md – Narrativa Estruturada**

Uma vez que planejamento está aprovado, você cria "framework de mensagens" que detalha especificamente como você vai falar sobre a campanha.

Este arquivo estrutura: narrativa geral (a grande ideia), pilares de mensagem (3-5 temas principais), pontos-chave por pilar, tom de voz, exemplos de como aplicar em diferentes canais.

**Por que este arquivo está aqui:**
Você pode ter equipe criativa, agências parceiras, ou múltiplas pessoas criando conteúdo. Sem framework de mensagens claro, cada pessoa vai estruturar diferente e mensagens vão se dispersar. Com framework, todos começam do mesmo ponto.

**Estágio do projeto:** Semana 8-9

**Como ele é usado:**
- Compartilhado com equipe criativa como "briefing criativo"
- Consultado quando alguém está criando post, vídeo, ou conteúdo
- Checklist: "este conteúdo está dentro do framework? Se não, remova-o ou justifique mudança."
- Validação com cliente antes de conteúdo público

**Exemplo real:** Para Aura Brews: "Pilar 1: Origem Ética. Cada post sobre este pilar mostra histórias de agricultores, sourcing, impacto. Tone: Inspirador mas genuíno, não corporativo."

---

**Nota Importante:** Arquivos como "ESG Campaign Objectives & KPIs.md" e "Documento de Planejamento de Campanha.md" existem em `/05_templates/Docs_complementares/` como templates reutilizáveis. Quando você inicia um projeto com campanha, copie esses templates para `/02_projects/` e preencha com dados reais.

## Como Este Diretório Diferencia de Outros

**vs `/05_templates/Docs_complementares/`:**
- Docs_complementares: Templates genéricos, virgens, prontos para copiar
- `/02_projects/`: Cópias daqueles templates, preenchidas com dados reais, em uso ativo

**vs `/00_meta/framework_rules/`:**
- Framework Rules: Define como criar documentos (padrões, schemas)
- `/02_projects/`: Exemplos reais de como documentos parecem quando preenchidos corretamente

**Relacionamento Real:**
```
Framework_rules/ (define HOW)
    ↓
Core-Docs/ (template virgins)
    ↓
02_projects/ (templates preenchidos em uso real)
```

## Como Usar Este Diretório

### Como Começar Novo Projeto

1. Vá para `/05_templates/Docs_complementares/`
2. Copie os templates que você precisa (Brand Profile Questionnaire, pesquisa, brief, planejamento, etc.)
3. Crie nova pasta em `/02_projects/` com nome do projeto
4. Cole templates naquela pasta
5. Comece preenchendo com dados reais
6. Use exemplos aqui como referência de "como deve parecer quando preenchido"

### Como Aprender a Usar Templates

1. Vire para `/02_projects/` (você está aqui)
2. Abra um dos 7 arquivos
3. Veja como foi preenchido
4. Use como modelo para seu próprio preenchimento

### Como Referência

- Se você não sabe como preencher campo em template, procure arquivo similar aqui
- Se você quer ver exemplo completo de "planejamento de campanha", abra o arquivo aqui
- Se você está treinando alguém novo, mostre estes arquivos como exemplos

## Boas Práticas ao Trabalhar Aqui

**✅ FAZER**
- Mantenha documentos atualizados conforme projeto evolui
- Revise KPIs contra desempenho real regularmente (mensal)
- Use estes documentos como base para comunicação com cliente
- Referencie estes documentos em reuniões para manter alinhamento
- Copie templates daqui para próximo projeto

**❌ NÃO FAZER**
- Não deixe documentos estagnados (se não atualizou em 1 mês, talvez projeto está parado)
- Não ignore KPIs que não são alcançados
- Não use padrão antigo de documentação quando você começa novo projeto aqui
- Não crie novo projeto sem usar template apropriado como base

## Manutenção e Evolução

Conforme projeto progride:

- **Mês 1:** Questionário → Pesquisa → Briefing → Planejamento (documentos estão sendo CRIADOS)
- **Mês 2-4:** Planejamento → Campanha → Mensagens → KPIs (documentos estão sendo EXECUTADOS)
- **Mês 5:** Status reports mensais são criados, referenciando estes documentos (documentos servem como BASE)
- **Mês 6:** Relatório final valida contra KPIs, resume aprendizados (documentos servem como REFERÊNCIA)

Ao final de projeto, estes 7 arquivos ficam como registro histórico completo: "então foi assim que fizemos este projeto."

## Conclusão: Seu Espaço de Trabalho Vivo

Diferente de `/05_templates/` que é biblioteca estática de templates, este diretório é **espaço de trabalho vivo**. Seus documentos estão aqui porque estão em uso. Eles são consultados, atualizados, referenciados.

Quando projeto termina, estes documentos migram para arquivo. Quando novo projeto começa, você cria novos arquivos aqui. Este ciclo contínuo é como times maduras gerenciam documentação: documentos vivos durante projeto, arquivados quando terminado, referenciados para aprender em próximos projetos.
